# Tenure v1 Architecture

## Overview

Tenure is a distributed lease coordination service for multitenant systems. It provides exclusive, time-bounded ownership of shared resources identified as `(tenant_id, resource_id)`. The service is implemented as a replicated state machine: mutating lease operations are serialized through consensus, producing one authoritative history for each resource.

v1 intentionally uses a **single shared Raft group** with **logical multitenancy**. This is a complexity-control decision. It allows the project to focus on correctness, lease semantics, and persistence boundaries before introducing partitioning and rebalancing.

The design nevertheless includes a placement abstraction from day one:

```text
placement(tenant_id, resource_id) -> raft_group_id
```

In v1, placement always returns the same group identifier. The API and internal model are written so future sharding can be added without redefining client-visible lease semantics.

## Goals

- Provide strongly consistent exclusive leases for `(tenant_id, resource_id)`.
- Support `Acquire`, `Renew`, `Release`, `GetLease`, and tenant-scoped `ListLeases`.
- Enforce first-class multitenancy in identity, authorization, quotas, and observability.
- Return fencing tokens to protect downstream systems from stale actors.
- Support idempotent client retries using request IDs.
- Persist enough local durable state per node for crash recovery.
- Preserve future sharding hooks without complicating v1 correctness.

## Non-Goals

- Shared, reader-writer, or priority leases.
- Atomic operations across multiple resources.
- Cross-tenant APIs or cross-tenant scans.
- Immediate multi-group deployment or online rebalancing.
- Detailed deployment manifests, production runbooks, or orchestration artifacts.
- Global fairness guarantees across every tenant under overload.
- Follower-served read paths in v1.

## Core use cases

- Singleton job execution within one tenant.
- Leader election for tenant-owned control loops.
- Ownership of externally mutable resources such as partitions, shards, or maintenance tasks.
- Tenant-scoped coordination where stale-writer protection is required.

## Why leases instead of simple distributed locks

A simple lock service is often defined by mutual exclusion alone: only one actor holds the lock at a time. A lease service must define more:

- how long ownership is valid
- who decides that a lease has expired
- what happens when the old owner keeps acting after logical expiry
- how clients safely retry requests
- how failover affects authority

Tenure therefore models time, expiration authority, fencing, and idempotency directly.

### Lease service vs. simple lock service

| Concern | Simple lock service | Tenure lease service |
| --- | --- | --- |
| Ownership duration | Often implicit or session-bound | Explicit TTL and expiry time |
| Stale-writer protection | Commonly absent | Required via fencing tokens |
| Retry semantics | Often ad hoc | Explicit idempotent request IDs |
| Failover semantics | Sometimes underspecified | Defined through replicated authoritative history |
| Tenant isolation | Optional | First-class |

## System model

Tenure is a bootstrap-safe replicated state machine driven by one Raft log in v1. It is treated as a foundational P0 / tier-0 coordination daemon that should be able to come up early on Linux hosts or VMs and support higher-level services rather than depending on them for its own correctness.

- **Replicas:** A set of self-contained node processes form one Raft group.
- **Leader:** The current Raft leader handles authoritative admission decisions for mutating operations and is the only replica that serves `GetLease` and `ListLeases` in v1.
- **Followers:** Followers replicate log entries and apply committed state-machine transitions. They do not independently mutate lease state because their local clocks think a lease has expired, and they do not serve authoritative reads in v1.
- **Clients:** Clients address resources as `(tenant_id, resource_id)` and include request IDs on mutating calls.
- **Storage:** Each node maintains its own durable Raft and state-machine persistence on local disk.
- **Bootstrap inputs:** Each node starts from local configuration, explicit node identity, local credentials if enabled, and statically configured peer transport endpoints.

Correctness dependencies in v1 are intentionally minimal: the OS process model, local disk, TCP/IP networking, local configuration files, and local credential or certificate material. The architecture does not require DNS, Kubernetes, an external database, a service mesh, or another coordination system to form or operate the cluster correctly.

## Deployment and runtime assumptions

The architecture captures a narrow operating model for v1 without turning this document into an ops runbook. The default mental model is direct-host deployment: one Tenure process per Linux host or VM, supervised by `systemd` or an equivalent local process manager.

| Assumption | v1 expectation |
| --- | --- |
| Host environment | Linux host or VM running a self-contained Tenure process. |
| Data durability | Writable local data directory for Raft state, snapshots, and materialized lease state. |
| Node identity | Static node ID defined in local configuration. |
| Peer bootstrap | Static peer list with concrete transport endpoints such as `10.0.0.12:7000`. |
| Credentials | Optional local TLS keypair / certificate material loaded from local files. |
| Process supervision | `systemd` or equivalent local supervision is the default operating assumption. |
| Containerization | Optional packaging choice only; not a correctness requirement and not the default design center. |

## Explicit non-dependencies

The following items may exist in some environments as conveniences around Tenure, but they are not correctness requirements and must not be startup prerequisites in v1.

| External system | v1 position |
| --- | --- |
| DNS | Not required for bootstrap, peer communication, or steady-state correctness. |
| Kubernetes | Not required for deployment, membership, or correctness. |
| Service mesh | Not required for peer communication or client correctness. |
| External service discovery | Not required; peer membership is statically configured in v1. |
| Remote config service | Not required; nodes start from local configuration. |
| External SQL/NoSQL database | Not required; authoritative state comes from the replicated log plus local durable storage. |
| Redis or cache service | Not required for leader routing, dedupe correctness, or lease state. |
| Another coordination service | Not required for bootstrap or correctness. |

## Networking and transport

Inter-node Raft communication uses TCP in v1. Peers communicate via statically configured concrete endpoints and should prefer long-lived connections so the transport path stays simple to reason about during bootstrap, failover, and recovery.

Rules for v1:

- Peer identity for correctness is anchored in explicit node IDs plus configured transport endpoints.
- Concrete endpoints should be configured as values such as `IP:port`; hostnames may exist as operator-friendly labels but are not required.
- UDP is out of scope for v1.
- Transport multiplexing is out of scope for v1.
- If multiplexing is introduced later, it must sit behind a stable transport abstraction and must not change lease semantics, leader rules, or the client API contract.

## DNS avoidance

DNS must not be required for cluster formation, peer communication, leader routing, or steady-state correctness in v1. A node should be able to start, connect to peers, participate in elections, and recover from restart using only local configuration that names peers by explicit node ID and concrete transport endpoint.

This means:

- correctness-critical peer identity does not depend on hostname resolution
- hostname-based discovery is not part of the v1 contract
- DNS-based leader routing is not part of the v1 contract
- optional human-friendly names may appear in logs or tooling, but they are not authoritative inputs to consensus behavior

## API surface

### Mutating operations

- **Acquire:** Attempt to create a new exclusive lease.
- **Renew:** Extend the expiry time of an active lease.
- **Release:** Voluntarily relinquish an active lease.

### Read operations

- **GetLease:** A leader-only linearizable read that returns the current lease view for one `(tenant_id, resource_id)`.
- **ListLeases:** A leader-only tenant-scoped paginated linearizable read in v1.

`GetLease` and `ListLeases` are leader-only in v1 because that is the simplest correct behavior, the easiest behavior to explain and test, and the least likely to return stale data during failover or leadership uncertainty. Safer follower-served reads, such as a quorum-backed or ReadIndex-style path, are future work and are not part of the v1 contract.

### Common response fields

Mutating success responses include:

- `lease_id`
- `expiry_time`
- `fencing_token`

These fields are part of the durable client contract.

## Lease lifecycle

1. A client calls `Acquire(tenant_id, resource_id, holder_id, ttl, request_id)`.
2. The leader checks authorization, tenant quota, TTL limits, idempotency history, and current lease state.
3. If admitted, the leader appends an acquire command to the Raft log together with the quota policy snapshot used for that admission decision.
4. After commitment, the state machine creates a new active lease, sets `expiry_time`, and advances the resource fencing token.
5. The client renews before expiry by sending `Renew(...)` with a new `request_id`.
6. If the holder releases, a release command transitions the lease to `RELEASED`.
7. If the holder does not renew in time, the leader may admit a later acquire after the lease has authoritatively expired.
8. A stale former holder may continue trying to act on the external resource, but downstream systems can reject it using the older fencing token.

## Data model

### Resource identity

The primary key is always:

```text
(tenant_id, resource_id)
```

There is no v1 API for a resource without tenant context.

### Lease record

A materialized lease record should contain at least:

| Field | Meaning |
| --- | --- |
| `tenant_id` | Tenant isolation key. |
| `resource_id` | Resource identity within the tenant. |
| `lease_id` | Unique identifier for the granted lease epoch. |
| `holder_id` | Client or logical actor that currently holds the lease. |
| `status` | `ACTIVE`, `EXPIRED`, `RELEASED`, or `ABSENT`. |
| `expiry_time` | Authoritative expiry timestamp computed from leader time at admission or apply. |
| `fencing_token` | Monotonic token for stale-writer protection. |
| `version` | Monotonic state-machine version for the record. |
| `created_time` | Lease creation time. |
| `last_renewed_time` | Most recent successful renewal time. |
| `released_time` | Time of release, if any. |
| `last_request_id` | Most recent applied mutating request ID, useful for auditing/debugging. |

### Idempotency record

Mutating requests should also create or update a deduplication record keyed by tenant-scoped request identity. A practical model is:

```text
(tenant_id, request_id) -> {operation, target_resource, result, applied_log_index, retention_deadline}
```

This avoids duplicate application on retries after leader changes or client timeouts.

## Raft integration model

### Replication boundary

Raft replicates **commands and ordered state transitions**, not copies of on-disk lease databases or filesystem contents.

Examples of replicated commands:

- `AcquireLease`
- `RenewLease`
- `ReleaseLease`
- optional administrative configuration changes

### Single-group v1 model

All lease mutations flow through one Raft group. This means every resource in v1 shares one authoritative ordered log, even though resources remain logically independent at the API layer.

Milestone 7 makes the implementation seam explicit without changing that default. Requests now pass through:

```text
API -> Router -> GroupRuntime -> LeaseService -> Raft/state machine
```

In the default deployment, the router still resolves every resource to one logical group. In local simulation mode, multiple logical groups may run in one process so routing, placement, and per-group recovery can be exercised deterministically without introducing live rebalancing or changing the external API.

### Leader-only admission decisions

The leader is responsible for admission decisions such as:

- whether an existing lease is still active
- whether a new acquire may proceed
- whether a renew is still valid
- whether quotas permit the request
- whether a request ID is a duplicate

Followers may replicate and apply committed entries, but they must not authoritatively invent new lease transitions solely from local clock observations.

### v1 read consistency model

v1 adopts one read rule for correctness and clarity:

- `GetLease` is a leader-only linearizable read.
- `ListLeases` is also leader-only in v1.
- Followers do not serve authoritative lease reads in v1, even if they appear caught up.
- Clients and front ends must route these reads to the current leader or follow a `NOT_LEADER` redirect.

This keeps the first implementation easy to reason about and test. It also avoids stale follower reads during failover, delayed apply, or term uncertainty. v1 does not include follower-served reads, read leases, or any implicit quorum shortcut. Future work may add a safer follower-served mechanism, but that requires an explicit protocol such as quorum-backed validation or a Raft `ReadIndex`-style check.

## Expiration and time semantics

Lease systems are sensitive to ambiguous time behavior. Tenure adopts the following rules in v1:

- A lease has a TTL and an associated `expiry_time`.
- `expiry_time` is computed from leader time when the command is admitted or applied, not from a client-supplied wall clock.
- The authoritative system decision about whether a lease may be superseded is leader-mediated.
- Followers do not independently change authoritative lease state only because their local clocks pass `expiry_time`.
- Lease validity is determined by authoritative replicated state plus leader-mediated expiry rules, not by client-local wall clock.
- TTL must be bounded by both global service policy and per-tenant policy.
- The service may reject TTLs that are too short to operate safely or renewal patterns that are pathologically aggressive.
- Clients are expected to renew before expiry with a safety margin rather than waiting until the apparent deadline.
- The implementation should centralize time calculations behind an explicit clock abstraction so expiration logic remains testable.

### Why follower-local expiration is insufficient

A follower may have clock skew, delayed apply progress, or stale leadership knowledge. If followers independently marked leases expired and admitted new owners, the system could violate exclusive ownership. Therefore, local clock observation may inform cache invalidation or diagnostics, but not authoritative mutation.

### Admission-time and apply-time implications

The implementation must use a consistent rule for how leader time contributes to `expiry_time`. Whether the code captures leader time exactly at command admission or exactly at state-machine apply, the rule must be explicit, centralized, and covered by tests. A Scala implementation should expose this through a small time interface rather than scattering direct wall-clock reads across handlers, state-machine code, and persistence code.

### Renewal expectations

A successful renewal extends the current lease from the leader's authoritative notion of time and returns the updated `expiry_time`. Clients should treat renewal as a time-sensitive operation, not as a best-effort background hint. Example TTL values may appear in tests or examples, but the architecture does not define universal defaults.

## Fencing token semantics

Fencing tokens are monotonically increasing per `(tenant_id, resource_id)` and advance on each successful new grant of ownership.

Rules:

- A successful `Acquire` returns the current grant's `fencing_token`.
- A `Renew` returns the same token as the current active lease.
- A later successful acquire after release or expiration gets a larger token.
- Downstream resource managers should reject commands carrying older tokens than the highest token they have already accepted.

Fencing is critical because lease expiry alone does not stop a paused or partitioned old holder from continuing to send writes.

## Idempotency model

Mutating operations require client request IDs.

Rules:

- `Acquire`, `Renew`, and `Release` are idempotent with respect to `(tenant_id, request_id)`.
- Replaying the same request against the same target should return the original logical result.
- Reusing the same `request_id` for a different operation or different resource is invalid.
- Deduplication records should survive leader changes and normal restart windows by being part of durable state.
- Transport retries do not create new semantics; they are expected to replay the same request until a definitive response is obtained or the client gives up.

## Multitenancy model

Multitenancy is first-class, not a naming convention.

- Every resource key includes `tenant_id`.
- Authorization is checked within tenant scope.
- Quotas are defined per tenant.
- Observability signals are taggable or partitionable by tenant.
- `ListLeases` is tenant-scoped; there is no v1 cross-tenant list API.

Logical multitenancy in one Raft group does not remove the need for explicit tenant isolation in code and operations.

## Authorization and tenant isolation model

v1 assumes some upstream identity layer can assert caller tenancy and permissions. Within Tenure, the service should enforce:

- callers can only mutate or read resources in their authorized tenant
- clients cannot supply one tenant in auth context and another in request payload
- admin or debug tooling must be explicit about tenant scope
- logs and metrics must avoid accidental cross-tenant data disclosure

A future implementation may add richer per-tenant RBAC, but the foundational rule is that cross-tenant access is disallowed.

## Quotas and fairness

v1 should support at least these per-tenant controls:

| Quota | Purpose |
| --- | --- |
| Request rate | Protect the shared control plane from noisy tenants. |
| Active lease count | Bound memory and state growth per tenant. |
| Max TTL | Prevent long-lived leases that degrade turnover and recovery. |

Fairness in v1 is best-effort rather than globally optimal. The system should avoid starvation-inducing policies where practical, but strict fairness guarantees are out of scope.

## Observability

Useful observability signals include:

- request counts and latencies by operation and tenant
- successful and rejected acquires
- renew success/failure rates
- quota rejections by tenant and reason
- leader changes and term transitions
- apply lag, commit lag, snapshot timings, and log growth
- active lease counts by tenant
- stale request or fencing-related rejection counters

Structured logs should include tenant and resource identifiers where safe, plus lease and request IDs for traceability.

## Persistence and crash recovery

Persistence must be described carefully.

### Raft persistence

Each node durably stores its own Raft data, typically including:

- current term
- `voted_for`
- write-ahead log / Raft log entries
- snapshots and snapshot metadata

This data is replicated logically through the Raft protocol, not by copying one node's filesystem to another.

### Materialized lease state persistence

Nodes may also persist materialized state-machine data such as:

- lease records
- fencing token counters
- deduplication / idempotency records
- indexes used for tenant-scoped listing

This persistence is likewise local to each node. The authoritative content is derived from the committed log and snapshots, not from shared disk.

### Recovery behavior

After restart, a node restores local durable Raft state, replays or installs snapshots, and reconstructs the materialized lease state machine. If a new leader is elected, it becomes the authority for future admission decisions.

## Snapshotting and log compaction

To bound replay time and storage growth, the service should support snapshotting the materialized lease state machine.

Snapshot contents should include:

- lease records
- fencing token counters
- deduplication records needed for retry safety
- pagination/index state required to reconstruct `ListLeases`

Compaction must preserve correctness. In particular, removing old log entries cannot discard deduplication or fencing information still needed to reject duplicate or stale requests.

## Failure scenarios and expected behavior

### Leader crash after appending but before replying

A command may commit even if the client times out. The client retries with the same `request_id`. The new leader returns the original result via idempotent deduplication.

### Follower clock runs fast

The follower may suspect a lease is expired, but it must not independently grant a new one. Only the leader's admitted and committed transition changes authoritative ownership.

### Old holder resumes after partition

Suppose holder A had fencing token 10, lost connectivity, and its lease later expired. Holder B then acquires the lease with token 11. If A resumes and tries to write to the external resource, the downstream system should reject token 10 once it has seen token 11.

### Node crash and restart

The restarting node recovers from local durable Raft and state-machine persistence. It does not require filesystem synchronization from peers.

### Read visibility during leadership change

Mutating operations may be temporarily unavailable. In v1, `GetLease` and `ListLeases` remain leader-only during and after leader transition; followers must not present stale reads as authoritative.

## Security considerations

- Enforce tenant identity consistently across auth context, request payload, and stored records.
- Treat request IDs and lease metadata as potentially sensitive operational data.
- Prevent tenant enumeration through error messages or list APIs.
- Apply rate limits and quotas to reduce abuse of the shared group.
- Protect persisted state because snapshots and logs contain tenant/resource mappings and operational metadata.

## Future sharding design hooks

The core routing abstraction is:

```text
placement(tenant_id, resource_id) -> raft_group_id
```

v1 behavior:

```text
placement(_, _) -> group-1
```

Current implementation shape:

- `Router` is an explicit, testable boundary between the API/service surface and per-group execution.
- `HashRouter` provides deterministic placement using a stable hash of `(tenant_id, resource_id)` and a configured ordered group set.
- `GroupRuntime` marks the per-group execution boundary and keeps Raft/state-machine execution independent per logical group.
- `ListLeases` remains tenant-scoped and may fan out across groups while preserving the same external contract.
- Single-group operation remains the default runtime mode; multi-group support is currently intended for local simulation and architectural validation rather than live shard management.

Design implications:

- Client APIs remain keyed by `(tenant_id, resource_id)` rather than physical placement.
- `ListLeases` remains tenant-scoped so it can later fan out to multiple groups for one tenant if needed.
- Correctness arguments should rely on per-resource authoritative history within one group, not on global scans over all resources in the system.
- Future sharding may partition by tenant, resource hash, or other policy without changing external lease semantics.

## Example sequence: acquire -> renew -> expiration -> stale client rejected

1. Tenant `acme` requests `Acquire(acme, scheduler/primary, worker-A, ttl=10s, request_id=req-1)`.
2. The leader admits the request, commits it, and returns:
   - `lease_id = lease-100`
   - `expiry_time = 2026-03-20T12:00:10Z`
   - `fencing_token = 57`
3. At `12:00:08Z`, `worker-A` sends `Renew(..., request_id=req-2)` and receives `expiry_time = 12:00:18Z`, token `57`.
4. `worker-A` pauses and misses the next renewal window.
5. After authoritative expiry, `worker-B` sends `Acquire(..., request_id=req-3)` for the same `(tenant_id, resource_id)`.
6. The leader commits the new grant and returns token `58`.
7. `worker-A`, still holding stale local state, attempts to write to the protected downstream system using token `57`.
8. The downstream system rejects the write because it has already accepted token `58`.

This sequence illustrates why expiry alone is not enough; fencing is required to protect against delayed or partitioned old holders.

## Open questions

- How long should deduplication records be retained relative to client retry windows and snapshot cadence?
- Should tenant quotas be fully replicated configuration, locally cached policy, or externally sourced with periodic refresh?
- What pagination/index strategy best preserves tenant-scoped list performance while staying shard-compatible?
- What holder identity model is most useful in practice: opaque client ID, workload identity, or structured principal?
